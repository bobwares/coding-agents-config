# af-be-build-implementation

## Purpose

Use this skill to execute backend application generation by:

1. copying a selected tech stack implementation scaffold from the App Factory to the target project;
2. generating domain modules, services, APIs, and persistence from the `dsl-be-ddd.yaml` specification.

This skill is the execution layer that follows the planning skill. It transforms the plan into working code.

## When to Use

Use this skill when:

1. a backend execution plan has been generated by `af-be-build-plan`;
2. a tech stack implementation has been selected from the App Factory;
3. the `dsl-be-ddd.yaml` has been approved and is ready for implementation;
4. the target project is ready to receive the generated code.

Do not use this skill when:

1. no execution plan exists;
2. no tech stack has been selected;
3. the DSL has not been reviewed or approved;
4. the user is asking for planning or DSL generation rather than implementation.

## Inputs

The skill requires these inputs:

1. **Selected Tech Stack Implementation** — A directory path from the App Factory implementations:
   - `~/gallery/app-factory/tech-stack-implementations/backend/<implementation-name>/`
   - Examples:
     - `container-typescript-nestjs`
     - `container-python-fastapi`
     - `container-java-springboot`
     - `serverless-typescript-lambda`

2. **Backend Domain DSL** — The `dsl-be-ddd.yaml` file containing:
   - bounded contexts
   - aggregates and entities
   - commands and workflows
   - domain events
   - API hints
   - persistence hints

3. **Target Project Path** — The destination directory for the generated backend application.

4. **Execution Plan** (optional but recommended) — The `spec-be-plan.md` from `af-be-build-plan`.

## Required Outputs

The skill produces:

1. A fully scaffolded backend project in the target directory
2. Generated domain modules based on the DSL
3. An implementation manifest at `ai/specs/implementation-manifest.yaml`

## Execution Procedure

### Phase 1: Scaffold Initialization

#### Step 1.1: Validate Inputs

Verify that:

1. the selected tech stack implementation directory exists;
2. the `dsl-be-ddd.yaml` file exists and is valid YAML;
3. the target project path is writable;
4. no conflicting files exist (or user has confirmed overwrite).

#### Step 1.2: Copy Implementation Scaffold

Copy the entire tech stack implementation to the target project:

```bash
APP_FACTORY_IMPL="$HOME/gallery/app-factory/tech-stack-implementations/backend"
IMPL_NAME="<selected-implementation>"
TARGET_PROJECT="<target-path>"

cp -R "$APP_FACTORY_IMPL/$IMPL_NAME/." "$TARGET_PROJECT/"
```

#### Step 1.3: Initialize Project Metadata

Update project-specific files:

1. `package.json` / `pom.xml` / `pyproject.toml` — Update name, version, description
2. `README.md` — Update with project-specific content
3. `.env.example` — Adjust for project requirements

### Phase 2: Domain Code Generation

For each bounded context in the DSL, generate:

#### Step 2.1: Module Structure

Create the module directory structure based on the tech stack conventions:

| Tech Stack | Module Pattern |
|------------|----------------|
| NestJS     | `src/<context>/` with `*.module.ts`, `*.controller.ts`, `*.service.ts` |
| FastAPI    | `src/<context>/` with `router.py`, `service.py`, `models.py` |
| Spring Boot | `src/main/java/.../context/` with `Controller.java`, `Service.java` |
| Lambda     | `src/functions/<context>/` with `handler.ts`, `service.ts` |

#### Step 2.2: Entity and DTO Generation

For each aggregate/entity in the DSL:

1. generate entity classes/interfaces;
2. generate request/response DTOs;
3. generate validation decorators/schemas;
4. generate mapper utilities if needed.

#### Step 2.3: Service Layer Generation

For each command/workflow in the DSL:

1. generate service methods;
2. implement business logic placeholders;
3. wire dependencies;
4. add observability hooks (logging, tracing).

#### Step 2.4: API Layer Generation

For each API hint in the DSL:

1. generate controller/router endpoints;
2. apply authentication/authorization decorators;
3. wire request validation;
4. wire response serialization.

#### Step 2.5: Persistence Layer Generation

For each persistence hint in the DSL:

1. generate repository interfaces;
2. generate ORM entities/models;
3. generate migration scripts if applicable;
4. wire persistence to services.

### Phase 3: Integration Wiring

#### Step 3.1: Module Registration

Register all generated modules with the application bootstrap:

| Tech Stack | Registration Pattern |
|------------|---------------------|
| NestJS     | Import modules in `app.module.ts` |
| FastAPI    | Include routers in `main.py` |
| Spring Boot | Component scan or explicit imports |
| Lambda     | Export handlers in `serverless.yml` or SAM template |

#### Step 3.2: Configuration Setup

Generate configuration for:

1. database connections;
2. external service integrations;
3. authentication providers;
4. observability backends.

### Phase 4: Validation

#### Step 4.1: Static Validation

Run:

1. TypeScript/Python/Java compilation check;
2. lint checks;
3. import resolution validation.

#### Step 4.2: Test Scaffold

Generate test scaffolds for:

1. unit tests per service;
2. integration test stubs;
3. e2e test templates.

### Phase 5: Implementation Manifest

Generate `ai/specs/implementation-manifest.yaml`:

```yaml
version: 1
kind: implementation-manifest
metadata:
  generatedAt: <ISO_TIMESTAMP>
  sourceStack: <implementation-name>
  sourceDsl: dsl-be-ddd.yaml
  targetProject: <target-path>
modules:
  - name: <context-name>
    files:
      - path: <file-path>
        type: module|controller|service|entity|dto|repository
        status: generated
boundedContexts:
  - name: <context-name>
    aggregates: <count>
    entities: <count>
    services: <count>
    endpoints: <count>
validation:
  compilation: pass|fail
  linting: pass|fail
  testsGenerated: <count>
notes:
  - <any generation notes or warnings>
```

## Tech Stack Specific Patterns

### NestJS (container-typescript-nestjs)

Use these skills for generation:
- `nestjs-crud-resource` — Generate CRUD modules
- `prisma-persistence` — Generate Prisma schema
- `nestjs-observability` — Add observability

Directory structure:
```
src/
  <context>/
    <context>.module.ts
    <context>.controller.ts
    <context>.service.ts
    dto/
      create-<entity>.dto.ts
      update-<entity>.dto.ts
    entities/
      <entity>.entity.ts
```

### FastAPI (container-python-fastapi)

Directory structure:
```
src/
  <context>/
    __init__.py
    router.py
    service.py
    models.py
    schemas.py
```

### Spring Boot (container-java-springboot)

Directory structure:
```
src/main/java/com/example/<context>/
  <Context>Controller.java
  <Context>Service.java
  <Entity>.java
  <Entity>Repository.java
  dto/
    <Entity>Request.java
    <Entity>Response.java
```

### Serverless Lambda (serverless-typescript-lambda)

Directory structure:
```
src/
  functions/
    <context>/
      handler.ts
      service.ts
      types.ts
```

## Constraints

You must follow these constraints:

1. Never overwrite existing files without explicit user confirmation.
2. Preserve any existing code in the target project when possible.
3. Follow the tech stack's established conventions and patterns.
4. Generate code that passes the stack's linting and compilation.
5. Include TODO comments for business logic that requires human implementation.
6. Do not generate mock data or hardcoded test values in production code.
7. Generate real validation based on DSL constraints.
8. Wire actual dependencies, not stubs.

## Forbidden Shortcuts

Do not:

1. generate only empty stub files;
2. skip persistence layer generation;
3. skip API layer generation;
4. generate code that violates the stack's conventions;
5. ignore DSL validation rules or constraints;
6. hard-code configuration values;
7. skip the implementation manifest.

## Quality Bar

A valid execution must:

1. produce a project that compiles/builds without errors;
2. include all bounded contexts from the DSL;
3. wire all modules correctly;
4. include validation on all DTOs;
5. include observability hooks;
6. generate the implementation manifest;
7. leave clear TODOs for complex business logic.

## Expected Response Style

When running this skill:

1. state the selected tech stack implementation;
2. state the DSL file being used;
3. state the target project path;
4. confirm scaffold copy;
5. generate modules per bounded context;
6. report validation results;
7. generate the implementation manifest;
8. summarize what was created.
