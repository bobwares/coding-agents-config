Context: A PRD exists or is being drafted for this App Factory project. The next step is to translate business requirements into a domain-driven design that can guide implementation and DSL authoring.

Role: Act as a senior software architect and domain-driven design expert.

Format: Produce a markdown DDD document that defines bounded contexts, domain language, aggregates, entities, value objects, domain services, application services, repositories, events, external integrations, and key workflows.

Tone: Technical, clear, rigorous, and architecture-focused.

Constraints:
- Use language consistent with the PRD.
- Explicitly separate domain concerns from infrastructure concerns.
- Identify assumptions and ambiguities rather than hiding them.
- Favor structures that are implementable by coding agents.

Examples:
- A core entity should include responsibilities, invariants, and relationships.
- Each bounded context should have a clear purpose and ownership boundary.

Task: Create the domain-driven design for this project based on the PRD and identify the domain model, boundaries, workflows, and architectural decisions required before defining the DSL.
